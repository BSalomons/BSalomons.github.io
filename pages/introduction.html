Introduction

The home page for the database (salomonscommercial.com/clientadmin which will now be referred to as the “Database”) has a menu running along the left hand side with several headings (groups) and then this same list of headings with each of their subgroups. You will only ever be dealing with the “Realty" group for editing listing and property information. 

Objects

Each of these subgroups are referred to as objects within the Database structure. Each object has its own set of fields and records. You can think of each object as an Excel spreadsheet, each record as a row in the spreadsheet, and each field as a column in the spreadsheet. For example:

In this example, the object is “Property”, the fields are the columns, and records are the rows.

Different objects can have relationships with each other. For example, a Property can be sold or leased therefore a Listing is attached to that Property. In this case, the Property and Listing objects are independent of each other and have what’s called a “many-to-many” relationship; meaning one Property can have multiple Listings (for sale or lease) or one Listing can have multiple Properties (industrial park) associated with it. 

Other objects have a “dependent relationship” and are referred to as a “parent-child” relationship. This means that the parent record needs to exist prior to any associated child records. For example, Buildings are a child object to the Property parent (you can’t have a Building without a piece of land first). Likewise, Premises is a child of the Building, and by extension a grandchild of the Property object. This relationship is referred to as a “one-to-many” relationship as one Property can have multiple Buildings but a Building can only have one Property associated with it (ditto with Premises). 

Other objects within the Database are Zoning, Uses, and Municipalities. For these, Zoning is a child of a Municipality. Rarely will you need to enter new records for these objects as they have likely already been established. 

Forms

When you are editing a record within the object, the page where you enter data is called a form. You are populating the fields with data for a specific record within the object.

Each record has a unique key that the Database automatically generates upon creation. No two records have the same unique key and you can’t edit this field. This is how the Database differentiates and references between records. When you attach a Listing to a Property record, the Database is assigning that unique key to the other. When you break that relationship (say by assigning a different property to the listing) the Database is reassigning the unique key. 

The objects within Salesforce, and by extension the Database, have Record Types associated with them. This allows the system to have different fields within the same object. For example, the Property object has “Industrial”, “Retail”, “Office”, etc. record types since there are certain fields that only apply to industrial. Same with In Contract: Sale or Lease. 

About This Checklist:

Best practice is to save often to avoid a bug surfacing and negating all the data input. Bugs do occur from time to time so if you are saving often you can usually troubleshoot which field caused the error and you won’t have wasted your time by losing the data that was put in. 

For the remainder of this checklist, the following will be used to help differentiate:

  • Underlined words refer to an object either within the Database or Salesforce
  • Capitalized words refer to a field name
  • Quotations will refer to a field value, a record type, or an example
  • “Brokerage” refers to Salomons Commercial
